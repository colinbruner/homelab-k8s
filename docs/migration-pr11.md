# PR #11 Migration Runbook

One-time cutover from the old `k8s/namespaces/*` layout to the new `k8s/platform/*` + `k8s/apps/*` layout.

## Why This Requires Manual Steps

The old `namespaces` ApplicationSet scans `k8s/namespaces/*`. Merging PR #11 deletes that directory. With `prune: true`, ArgoCD would delete every Application it generated -- and each Application would prune its managed Kubernetes resources. The new ApplicationSets (`platform`, `apps`) live in `bootstrap/root/` and must be applied manually.

Additionally, the old `namespaces` ApplicationSet is itself managed by the `argocd` Application it generates (circular dependency). When the ApplicationSet prunes the `argocd` App, the App's managed resources -- including the ApplicationSet itself -- would also be deleted.

**If you just merge and walk away, ArgoCD will delete everything it manages.**

## What Changes

### Directory renames

| Old (k8s/namespaces/) | New location | App name change |
|------------------------|--------------|-----------------|
| argocd | k8s/apps/argocd | same |
| backup-documents | k8s/apps/backup-documents | same |
| backup-photos | k8s/apps/backup-photos | same |
| beszel | k8s/apps/beszel | same |
| cert-manager | k8s/platform/cert-manager | same |
| cloudflared | k8s/apps/cloudflared | same |
| crossplane-system | k8s/platform/crossplane | **crossplane-system -> crossplane** |
| csi-nfs | k8s/platform/csi-nfs | same |
| garage | k8s/apps/garage | same |
| gateway-system | k8s/platform/gateway | **gateway-system -> gateway** |
| metallb-system | k8s/platform/metallb | **metallb-system -> metallb** |
| ollama | k8s/apps/ollama | same |
| sftp | k8s/apps/sftp | same |

### New components (not previously ArgoCD-managed)

| Component | Location | Notes |
|-----------|----------|-------|
| 1password | k8s/platform/1password | Was bootstrap-only (helm install). Now ArgoCD-managed via helmCharts. |
| storage | k8s/platform/storage | Was k8s/cluster/storage, applied manually. Now ArgoCD-managed. |

### Namespace routing

The ApplicationSet sets `destination.namespace: {{.path.basename}}` but this is overridden by explicit `namespace:` in each kustomization.yaml or resource metadata. Actual resource namespaces do not change:

- `crossplane` dir -> resources still deploy to `crossplane-system` (kustomization `namespace: crossplane-system`)
- `gateway` dir -> resources still deploy to `gateway-system` and `envoy-gateway-system` (explicit metadata on each resource)
- `metallb` dir -> resources still deploy to `metallb-system` (kustomization `namespace: metallb-system`)

### ApplicationSet changes

| Old | New |
|-----|-----|
| 1 ApplicationSet: `namespaces` scanning `k8s/namespaces/*` | 2 ApplicationSets: `platform` scanning `k8s/platform/*`, `apps` scanning `k8s/apps/*` |
| Managed by the `argocd` Application (circular) | Managed manually via `kubectl apply -k bootstrap/root` |
| No sync-wave ordering | Platform apps ordered by sync-wave (-30 to 0) |

## Pre-Merge Steps

### Step 1: Verify cluster access

```bash
kubectl cluster-info
kubectl get applicationsets -n argocd
kubectl get applications -n argocd
```

Confirm you see the `namespaces` ApplicationSet and the 13 Applications it generates.

### Step 2: Record current state

Save the current state so you can verify nothing was lost after cutover.

```bash
# Snapshot current Applications
kubectl get applications -n argocd -o wide > /tmp/pre-migration-apps.txt

# Snapshot key workloads
kubectl get deployments --all-namespaces > /tmp/pre-migration-deployments.txt
kubectl get statefulsets --all-namespaces > /tmp/pre-migration-statefulsets.txt
kubectl get pv > /tmp/pre-migration-pvs.txt
```

### Step 3: Pause ArgoCD auto-sync

Prevent ArgoCD from acting on changes while we reconfigure.

```bash
# Scale down the application controller so it cannot sync/prune anything
kubectl -n argocd scale deploy argocd-applicationset-controller --replicas=0

# Verify it's down
kubectl -n argocd get deploy argocd-applicationset-controller
```

This is the safest approach -- with the controller stopped, ArgoCD cannot prune or sync regardless of what happens in git.

### Step 4: Delete the old ApplicationSet (non-cascading)

Remove the old `namespaces` ApplicationSet without deleting the Applications it generated. The Applications become orphans (still on the cluster, still tracking resources, but no longer generated by an ApplicationSet).

```bash
# Check the current ApplicationSet's preserveResourcesOnDeletion policy
kubectl get applicationset namespaces -n argocd -o yaml | grep -A5 'syncPolicy'

# Delete the ApplicationSet only -- Applications persist as orphans
kubectl delete applicationset namespaces -n argocd
```

> **Note:** As of ArgoCD v2.8+, `kubectl delete applicationset` with the default policy preserves the generated Applications. If your ArgoCD version is older, add `--cascade=orphan` or set `.spec.syncPolicy.preserveResourcesOnDeletion: true` on the ApplicationSet before deleting it.

Verify the Applications still exist:

```bash
kubectl get applications -n argocd
# Should still show all 13 Applications
```

### Step 5: Delete the orphaned cronhealth Application

The old `application-cronhealth.yaml` was managed by the `argocd` Application. The new one lives in `bootstrap/root/`. Delete the orphan so the new one can take its place.

```bash
kubectl delete application cronhealth -n argocd
```

## Merge

### Step 6: Merge PR #11

Merge the PR via GitHub. The controller is scaled down, so nothing happens on the cluster yet.

```bash
gh pr merge 11 --merge
```

## Post-Merge Steps

### Step 7: Update ArgoCD installation

The new branch restructures ArgoCD's own configuration (argo-cd chart 9.4.3, `--enable-helm` via `configs.cm`). Apply it before the controller comes back.

```bash
git checkout main && git pull

# Rebuild and apply ArgoCD from the new bootstrap layout
kustomize build --enable-helm bootstrap/argocd | kubectl apply --server-side --force-conflicts -f -

# Wait for the new ArgoCD to roll out
kubectl -n argocd rollout status deploy/argo-cd-argocd-repo-server --timeout=300s
kubectl -n argocd rollout status deploy/argo-cd-argocd-server --timeout=300s
```

### Step 8: Apply the new ApplicationSets

```bash
kubectl apply -k bootstrap/root
```

This creates:
- `platform` ApplicationSet (scans `k8s/platform/*`)
- `apps` ApplicationSet (scans `k8s/apps/*`)
- `cronhealth` Application

### Step 9: Scale the controller back up

```bash
kubectl -n argocd scale deploy argo-cd-argocd-application-controller --replicas=1

# Wait for it to start
kubectl -n argocd rollout status deploy/argo-cd-argocd-application-controller --timeout=120s
```

### Step 10: Handle Application name collisions

The new ApplicationSets will try to create Applications with the same names as some orphaned ones (argocd, backup-documents, backup-photos, beszel, cert-manager, cloudflared, csi-nfs, garage, ollama, sftp). ArgoCD will adopt or error on these depending on version.

**If ArgoCD adopts them** (updates source path in-place): the Applications now point at the new paths. Verify:

```bash
kubectl get applications -n argocd -o custom-columns='NAME:.metadata.name,PATH:.spec.source.path,STATUS:.status.sync.status'
```

**If ArgoCD errors** (cannot create, name already exists): delete the conflicting orphan Applications so the ApplicationSet can recreate them. Delete one at a time and confirm the new one appears:

```bash
# For each conflicting app (example: cert-manager)
kubectl delete application cert-manager -n argocd
# The ApplicationSet immediately recreates it pointing at k8s/platform/cert-manager
kubectl get application cert-manager -n argocd -o yaml | grep path
```

### Step 11: Clean up renamed Applications

Three old Applications have no name collision and will remain as orphans. Delete them -- their resources are now managed by the new Applications under the new names:

```bash
kubectl delete application crossplane-system -n argocd
kubectl delete application gateway-system -n argocd
kubectl delete application metallb-system -n argocd
```

Verify the new Applications exist and are syncing:

```bash
kubectl get application crossplane -n argocd
kubectl get application gateway -n argocd
kubectl get application metallb -n argocd
```

### Step 12: Verify new platform components

1password and storage were not previously ArgoCD-managed. Verify they sync correctly:

```bash
# 1password -- was bootstrap-only, now ArgoCD-managed
kubectl get application 1password -n argocd
kubectl get pods -n 1password

# storage -- was manually applied, now ArgoCD-managed
kubectl get application storage -n argocd
kubectl get pv
```

### Step 13: Verify Crossplane version

The new chart pins Crossplane to 2.3.2 (app version 2.x). If the cluster currently runs Crossplane 1.x, this is a **major upgrade** with potential breaking changes.

```bash
# Check current Crossplane version
kubectl get deploy crossplane -n crossplane-system -o jsonpath='{.spec.template.spec.containers[0].image}'
```

If you see a 1.x image, review the [Crossplane 2.0 migration guide](https://docs.crossplane.io/latest/guides/migration-from-v1/) before allowing the sync. You may need to pause the `crossplane` Application while you prepare.

## Verification

### Step 14: Full cluster health check

```bash
# All Applications should be Synced + Healthy
kubectl get applications -n argocd -o custom-columns='NAME:.metadata.name,SYNC:.status.sync.status,HEALTH:.status.health.status'

# Compare deployments against pre-migration snapshot
kubectl get deployments --all-namespaces > /tmp/post-migration-deployments.txt
diff /tmp/pre-migration-deployments.txt /tmp/post-migration-deployments.txt

# Compare statefulsets
kubectl get statefulsets --all-namespaces > /tmp/post-migration-statefulsets.txt
diff /tmp/pre-migration-statefulsets.txt /tmp/post-migration-statefulsets.txt

# PVs should all still be Bound
kubectl get pv

# Backup jobs should be running
kubectl get pods -n backup-documents
kubectl get pods -n backup-photos

# Gateway should be serving traffic
kubectl get gateway -n gateway-system
kubectl get httproute --all-namespaces
```

### Step 15: Verify no orphaned resources

```bash
# No old ApplicationSet should remain
kubectl get applicationset -n argocd
# Should show: platform, apps (no 'namespaces')

# No orphan Applications from old layout
kubectl get applications -n argocd --show-labels | grep -v 'applicationset'
# Only cronhealth should appear (it's a standalone Application, not from an ApplicationSet)
```

## Rollback

If something goes wrong at any step:

**Before merge (Steps 1-5):**
```bash
# Re-apply the old ApplicationSet from main
kubectl apply -f <(git show main:k8s/namespaces/argocd/resources/applicationset.yaml)

# Scale controller back up
kubectl -n argocd scale deploy argo-cd-argocd-application-controller --replicas=1
```

**After merge (Steps 6+):**
```bash
# Scale controller down immediately
kubectl -n argocd scale deploy argo-cd-argocd-application-controller --replicas=0

# Revert the merge on GitHub or locally
git revert <merge-commit> && git push

# Delete new ApplicationSets
kubectl delete applicationset platform apps -n argocd

# Re-apply old ApplicationSet
kubectl apply -f <(git show main:k8s/namespaces/argocd/resources/applicationset.yaml)

# Scale controller back up
kubectl -n argocd scale deploy argo-cd-argocd-application-controller --replicas=1
```
